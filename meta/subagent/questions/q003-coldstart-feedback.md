# Q003 Cold-start User Feedback

用于在 Docker 冷启动中还原“opencode agent 代表使用者”的反馈链路，服务产品决策，而非技术自测。

## 策略边界

- 本轮聚焦：边界理解、安装可达、首次价值路径、失败恢复可解释性。
- 本轮不做：弱网场景验证、探索额度治理机制。

## 问题集（高杠杆版）

### 认知阶段

1. 你会如何一句话解释 cowork-cli 的价值？
2. 它和 `git clone` 的核心差异是什么？【边界偏差高敏】
3. 你预期 10 分钟内拿到什么“首个有效结果”？

### 安装阶段

4. 安装流程里最不放心的点是什么？
5. `self-update` 会更新什么、不会更新什么？【边界偏差高敏】
6. 你如何确认“安装成功且可用”？

### 首次执行阶段

7. `clone init` 后你预期下一步是什么？
8. 首次输出里哪 1-2 行最有用？
9. 你如何确认没有污染本地状态？【边界偏差高敏】

### 失败恢复阶段

10. 首次失败后你会先看哪里？
11. `clone version` 远端失败时你期望什么下一步提示？
12. `clone preview` session 已存在时，它在保护什么？【边界偏差高敏】

### 边界理解阶段

13. 你认为哪些动作应由 agent 自动做，哪些必须显式确认？【边界偏差高敏】
14. 哪些场景你认为不该使用 cowork-cli？【边界偏差高敏】
15. 你能说清它改哪些状态、如何回滚吗？【边界偏差高敏】

## 判定锚点

- `PASS`: 能稳定复述边界（协作接口层）、首个价值路径清晰、失败后有可执行下一步。
- `WARN`: 可完成任务但边界认知模糊，或失败恢复依赖人工解释。
- `FAIL`: 关键边界持续误解（如把 cowork 当业务规则引擎），或失败后无法自救。
